# systemg — Security Review (2026-07-13)

Process supervisor / init system. Trust model: the **config file is operator-controlled** (commands, users, health-check URLs there are intended execution, not vulns). The real attack surface is the **control socket** (`control.sock`), the IPC protocol, filesystem permissions, and every cross-privilege boundary (privilege dropping, capabilities, namespaces, cgroups, FD inheritance, signal handling, PID reuse, env sanitization).

This review follows up on `reports/2026-06-11.md`. The headline result: **every finding from the prior audit (P0–P5, K0–K2) is fixed or is an explicitly accepted risk** — the control socket now enforces `SO_PEERCRED`/`getpeereid` peer authentication, config paths loaded over the socket are validated with `ensure_trusted_config`, runtime dirs/files/socket are created `0700`/`0600`, the empty-command panic and the integer-overflow/env-var panics are gone, supplementary groups are always reset on a uid switch, and activation FDs are marked `FD_CLOEXEC`.

The new findings below are second-order: they live **behind** the now-correct authentication boundary, so under the same-UID trust model none is a remote or cross-privilege break. They are reported as hardening / defense-in-depth, because the socket is the trust boundary and its inputs (service names, control-line length) are still treated as free-form. Findings are ordered lowest → highest.

### P5 — `serde_yaml` is an unmaintained dependency

**What.** All config parsing goes through `serde_yaml`, whose author archived the crate and marked the final `0.9.34` release `+deprecated`.

**Where.** `Cargo.toml:41` (`serde_yaml = "0.9"`); `Cargo.lock` resolves `serde_yaml 0.9.34+deprecated`. Used at `config.rs` (`serde_yaml::from_str` on config content).

**Why.** An unmaintained crate on the primary parse path receives no security fixes ([RUSTSEC-2024-0320](https://rustsec.org/advisories/RUSTSEC-2024-0320), unmaintained). Config is operator-controlled so there is no direct attacker input here, but a future YAML-parser bug (billion-laughs-style alias expansion, deep-nesting stack exhaustion) would never be patched upstream. Same class the prior report tracked for `rustls-pemfile` in `reports/CVE.md`.

**Fix.** Migrate to a maintained YAML parser (`serde_yaml_ng` or `serde_yml`), or move to a maintained format. No API break for the current usage. Add `serde_yaml` to the RUSTSEC deny-list once migrated.

### P4 — Unbounded control-line read from the socket

**What.** The supervisor reads a control command with `BufReader::read_line` into a `String` with no length cap. A single connection that streams bytes without a newline grows the buffer until the process is OOM-killed.

**Where.** `src/ipc.rs:410` (`read_command`) — `reader.read_line(&mut line)?` with no `take()`/limit; the same pattern in `send_command`/`send_command_with_timeout` on the client side.

**Why.** Unbounded allocation driven by socket input ([CWE-770](https://cwe.mitre.org/data/definitions/770.html), allocation without limits). Now gated behind `authenticate_peer`, so only the supervisor's own UID or root can trigger it — a same-UID local DoS, not a cross-privilege break. Still worth bounding: the control channel should never let one framed message exhaust supervisor memory.

**Fix.** Wrap the reader in `.take(MAX_CONTROL_LINE)` (e.g. 1 MiB) and return a framing error past the cap; reject over-long messages before deserialization.

### P3 — Config check→load TOCTOU in `ensure_trusted_config`

**What.** `ensure_trusted_config` `stat`s the path and rejects group/other-writable or foreign-owned files, then the caller re-opens the same path by string in `load_config`. Between the check and the open the path (or a directory component) can be swapped, so the file validated is not necessarily the file parsed and executed.

**Where.** `src/runtime.rs:182-210` (`ensure_trusted_config` uses `fs::metadata`, which follows symlinks); callers at `src/supervisor.rs:657-659`, `697-698`, `1801-1802`, `2111-2112`, `2213-2214` do `canonicalize().unwrap_or(resolved)` then check then `load_config(...)`.

**Why.** Classic time-of-check/time-of-use race ([CWE-367](https://cwe.mitre.org/data/definitions/367.html)) plus symlink-following in the check ([CWE-59](https://cwe.mitre.org/data/definitions/59.html)). Exploiting it requires write access to a directory component of the config path and same-UID authentication to reach the load path, so it is a narrow, authenticated race rather than a general privesc — but a config that passed validation could still be replaced with an attacker's before it is read and its `command:` entries are launched with the supervisor's privileges. Compare [CVE-2021-3560](https://nvd.nist.gov/vuln/detail/CVE-2021-3560)-style check/act gaps.

**Fix.** Open the file once (`O_NOFOLLOW` on the final component), validate the `fstat` of the *open descriptor*, then parse from that same descriptor — never re-resolve the path string between check and use.

### P2 — Dropped services inherit the full supervisor environment

**What.** When launching a service, systemg sets the configured env vars on top of the inherited process environment and only removes a small session-scoped denylist (`SSH_AUTH_SOCK`, `SSH_AGENT_PID`, `SSH_CLIENT`, `SSH_CONNECTION`, `SSH_TTY`). It never calls `Command::env_clear()`, so a service dropped to a lower-privileged user still receives the supervisor's entire (root) environment.

**Where.** `src/daemon.rs:2211` (`collect_service_env` merged onto the inherited env), `2235-2247` (only `SESSION_SCOPED_ENV_VARS` / configured strip-list removed); constants at `src/constants.rs:135`.

**Why.** Environment leak across a privilege boundary ([CWE-526](https://cwe.mitre.org/data/definitions/526.html) / [CWE-200](https://cwe.mitre.org/data/definitions/200.html)): any secret, token, or path in root's environment (e.g. `AWS_SECRET_ACCESS_KEY`, `VAULT_TOKEN`, a poisoned `PATH`/`LD_*`) is handed to every dropped-privilege child that was never meant to see it. systemd deliberately starts services from a clean environment for exactly this reason. Operator-config governs which vars are *added*, so this is a hardening gap, not arbitrary injection.

**Fix.** Default to a clean environment for services that drop privileges: `env_clear()` then install only the resolved (config + `PATH`/`HOME`/`USER`) set, with an opt-in `inherit_env` for the rare service that needs it. At minimum, extend the default strip-list to the well-known secret-bearing and loader (`LD_*`) variables.

### P1 — `Spawn` children run with no privilege drop

**What.** The dynamic `Spawn` control command builds a child with a bare `std::process::Command::new(program)` and no `PrivilegeContext` / `pre_exec` hook, so — unlike configured services — spawned children never get uid/gid switch, capability trimming, rlimits, namespaces, or cgroup placement. In System mode they run as **root**.

**Where.** `src/supervisor.rs:1932-1960` (`handle_spawn`): no `PrivilegeContext::from_service`, no `apply_pre_exec`, no user switch — contrast the configured-service path at `src/daemon.rs:2214-2252`.

**Why.** Missing privilege drop on a spawn path ([CWE-250](https://cwe.mitre.org/data/definitions/250.html), execution with unnecessary privileges). Now that `authorize_spawn` requires `parent_pid` to map to a real tracked service and the socket is authenticated, only a same-UID/root peer can reach this — so it is not a cross-privilege break under the trust model. But it is a sharp edge: dynamically spawned helpers silently run at maximum privilege regardless of the parent service's own hardening, defeating the expectation that a child of a dropped service is at most as privileged as its parent.

**Fix.** Resolve the parent service's `PrivilegeContext` (via the authorized `root_service`) and apply the same `pre_exec` drop sequence to spawned children, or at minimum inherit the parent's user/caps so a spawn cannot exceed its authorizing service's privilege.

### P0 — Path traversal via socket-supplied service name in the log subsystem

**What.** The `Logs` control command carries a client-chosen `service` string. When it does not match a known unit, the supervisor still builds on-disk log paths directly from that raw string and (a) reads matching files back over the socket and (b) **creates** files/directories for it. The name is never validated against `..` or path separators, so a name like `../../../../etc/cron.d/pwn` escapes the log directory.

**Where.**
- `src/supervisor.rs:1607-1621` — on no unit match, the raw `service_name` flows into `get_service_log_path` / `resolve_log_path` and then `stream_log_to_socket`.
- `src/logs.rs:613-624` — `canonical_log_path` / `canonical_combined_log_path` do `runtime::log_dir().push(format!("{service}_{kind}.log"))` with the raw name (no sanitization; `normalize()` is applied only in the `locate_existing_*` matchers, not to the constructed path).
- `src/logs.rs:1120-1136` — `resolve_tail_targets` calls `touch_log_file(&stdout_path)` / `(&stderr_path)`, and `src/logs.rs:885-891` `touch_log_file` does `fs::create_dir_all(parent)` + `OpenOptions::create(true).open(path)`.

**Why.** Relative-path traversal from untrusted socket input ([CWE-22](https://cwe.mitre.org/data/definitions/22.html), with the write side [CWE-73](https://cwe.mitre.org/data/definitions/73.html), external control of file name/path). As the supervisor (root in System mode) this yields:
1. **Arbitrary file read** of any `*.log` / `*_stdout.log` / `*_stderr.log` path the supervisor can reach, streamed back to the caller.
2. **Arbitrary file/directory creation** (empty files, and whole parent trees via `create_dir_all`) at attacker-influenced paths ending in `.log` — e.g. seeding a `.log`-suffixed drop file or creating directories to influence other components.

It is gated behind `authenticate_peer`, so only the supervisor's own UID or root can send it — under the stated same-UID trust model this is not a privilege escalation. But the socket *is* the trust boundary, the service name is free-form attacker-controlled input crossing it, and a traversal-to-root-filesystem primitive should not exist even for authenticated peers. Compare [CVE-2021-33503](https://nvd.nist.gov/vuln/detail/CVE-2021-33503)-class path handling and the general init-system log-path hardening rationale.

**Fix.** Validate the service name at the socket boundary before it touches the filesystem: reject any name containing `/`, `\0`, or a `.`/`..` component (or accept only `[A-Za-z0-9._-]` and a bounded length), and additionally confirm every resolved log path stays inside `runtime::log_dir()` (canonicalize the parent and assert the prefix) before opening or `touch`-ing it. The unmatched-service branch should not create files for names that correspond to no known unit at all.

## Notes on areas reviewed and found sound

- **Privilege-drop ordering** (`privilege.rs:203-214`): `isolation → limits → nice → affinity → caps-pre → uid/gid switch → caps-post` is correct; `setgroups` precedes `setgid`/`setuid`.
- **Signal `killpg` guard** (`daemon.rs:1843`): `target_pgid == 0` is covered because it equals the supervisor's own pgid and is rejected by the `!= supervisor_pgid` check.
- **Dependency posture**: prior CVE bumps are present in `Cargo.lock` — `bytes 1.12.0`, `rand 0.8.6`, `reqwest 0.12.28` (drops the unmaintained `rustls-pemfile 1.x`). Only `serde_yaml` (P5) remains flagged.
